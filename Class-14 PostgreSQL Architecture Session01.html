Basic structure of a table on disk

If you have a table called students, inside the database’s folder (like base/16384/), you’ll see something like this:

16384/12345       ← main data file (actual rows)

16384/12345_fsm   ← free space map

16384/12345_vm    ← visibility map

 

🧩 1. fsm — Free Space Map

Purpose: Tracks how much free space is available in each page (8KB block) of the table.

📖 Why it exists:

When PostgreSQL inserts a new row, it needs to know where there’s room.
Instead of scanning the whole table file, it looks at the fsm to quickly find pages that have enough space.

Example:Let’s say your table’s data file (12345) has 4 pages:

Page 1 → 2000 bytes free 

Page 2 → 1000 bytes free 

Page 3 → 0 bytes free 

Page 4 → 4000 bytes free

The FSM will store a small summary:

[Page 1: 25% free]

[Page 2: 12% free]

[Page 3: 0% free]

[Page 4: 50% free]

So when PostgreSQL needs to insert a row of 1500 bytes,
it checks FSM → finds Page 4 → inserts row there directly.
No need to scan every page!

📦 Key point:

  • FSM = a map of empty spaces in data pages.
  • Used by INSERT and UPDATE to reuse space efficiently.
  • Helps reduce table bloat.

🪞 2. vm — Visibility Map

Purpose:Tracks which pages contain only visible (fully committed) tuples.
If all tuples on a page are visible to everyone, PostgreSQL marks it in the VM.

📖 Why it exists:

The visibility map helps PostgreSQL skip pages during VACUUM and ANALYZE.
If a page is fully visible (no dead rows), there’s no need to scan it again.

Example:Imagine this:

Page 1 → All tuples committed → visible

Page 2 → Has some deleted rows → not visible

Page 3 → All tuples committed → visible

The VM will mark:

[Page 1: visible]

[Page 2: not visible]

[Page 3: visible]

When VACUUM runs, it will skip Page 1 and 3, saving tons of time.
It only scans pages that might need cleanup.

📦 Key point:

  • VM = map of which pages are safe to skip during vacuum.
  • Used by VACUUM and index-only scans.
  • Greatly improves performance for large tables.
